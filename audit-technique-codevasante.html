1Objectif du document

Ce document constitue un audit technique complet du frontend React du projet CodevaSanté, réalisé après le merge de la Pull Request #11 sur la branche main.

L'objectif est d'identifier tous les problèmes bloquants, graves et mineurs présents dans le code actuel avant de continuer le développement du parcours patient (Sprint 1) et, a fortiori, avant de passer aux interfaces médecin, réceptionniste et admin.

Contrairement à une revue de code superficielle, cet audit examine chaque fichier source, vérifie la cohérence entre les couches (types → services → Redux → composants), et identifie les violations de la méthode Ahmed Yahya décrite dans le Guide de Prompts IA Générative.

⚠ Avertissement Plusieurs problèmes identifiés dans ce rapport rendent l'application non fonctionnelle au-delà d'un seul chargement de page. Ces points doivent être corrigés avant toute démonstration ou déploiement sur Vercel.

2Résumé exécutif

L'analyse du code source révèle 2 problèmes bloquants, 4 problèmes graves et 7 problèmes mineurs répartis sur 17 fichiers examinés.

Le problème le plus critique est l'absence totale de persistance de l'authentification. Le token JWT d'accès et le token de rafraîchissement sont stockés uniquement en mémoire Redux. À chaque rechargement de page, l'état Redux est réinitialisé et l'utilisateur se retrouve déconnecté. Cela rend le parcours patient inutilisable en conditions réelles.

Le second problème bloquant est l'utilisation dans PatientLayout.tsx de clés de traduction inexistantes dans translations.ts (navPatientHome, navPatientProfile, navPatientDoctors, navPatientAppointments, navLogout). TypeScript strict étant activé, le build de production échoue.

Par ailleurs, la méthode de développement par phases d'Ahmed Yahya a été partiellement violée : des pages patient complètes ont été générées et committées avant que les services API aient été validés avec de vrais tokens JWT en Postman, introduisant un risque de divergence silencieuse entre les types frontend et les réponses backend réelles.

3Audit fichier par fichier

src/lib/axios.ts ✓ CORRECT

C'est le fichier le mieux écrit du projet. La logique de rafraîchissement silencieux est implémentée correctement : utilisation de axios.post natif (et non apiClient) pour éviter la récursion infinie dans l'intercepteur, file d'attente pour les requêtes concurrentes en attente de token, logout forcé uniquement si le refresh échoue.

Problème mineur — Duplication de BASE_URL

La constante BASE_URL est définie ici ET dans authService.ts : 

// axios.ts
const BASE_URL = import.meta.env.VITE_API_URL ?? 'https://codevasante.pythonanywhere.com/api'

// authService.ts — copie exacte
const BASE_URL = import.meta.env.VITE_API_URL ?? 'https://codevasante.pythonanywhere.com/api'

Deux sources de vérité pour la même valeur. Si l'URL de production change, l'un des deux fichiers sera oublié. Correction : exporter BASE_URL depuis axios.ts et l'importer dans authService.ts.

src/features/auth/authSlice.ts 🔴 BLOQUANT
🔴 Bug critique — Aucune persistance de l'authentification Le token JWT vit uniquement en mémoire Redux. Chaque rechargement de page réinitialise l'état à initialState. L'utilisateur est systématiquement déconnecté. L'application est inutilisable en conditions réelles. 
const initialState: AuthState = {
  user: null,
  token: null,        // ← perdu au refresh
  refreshToken: null, // ← perdu au refresh
  isAuthenticated: false,
}

Aucune lecture ni écriture dans localStorage ou sessionStorage. Aucun redux-persist configuré dans le store. Le comportement observé est le suivant :

  1. L'utilisateur se connecte → setCredentials est appelé → state Redux peuplé.
  2. L'utilisateur navigue vers une page patient → fonctionne (Redux est en mémoire).
  3. L'utilisateur rafraîchit la page → Redux se réinitialise → isAuthenticated: false.
  4. ProtectedRoute détecte l'état non authentifié → redirige vers /app/login.
  5. L'utilisateur est renvoyé à la page de connexion à chaque rechargement.

Correction requise : implémenter la persistance dans le slice via localStorage (load à l'initialisation, save dans chaque reducer concerné) ou installer redux-persist avec une configuration blacklist pour les données sensibles.

src/features/patient/patientSlice.ts 🟠 GRAVE

La structure générale est correcte. Les thunks createAsyncThunk sont bien séparés, les matchers isPending et isRejected réduisent la duplication. Néanmoins trois problèmes sont identifiés.

Problème 1 — cancelAppointment.fulfilled ne met pas à jour la liste

.addCase(cancelAppointment.fulfilled, (state) => {
  state.loading = false
  // list re-fetch required — no GET /appointments/me/ endpoint yet
})

Quand un patient annule un rendez-vous, la liste Redux n'est pas mise à jour. L'UI ne reflète pas le changement. Sans GET /appointments/me/, une re-fetch n'est pas possible, mais au minimum le statut du RDV annulé devrait être mis à jour localement par son id.

Problème 2 — Pages générées avant validation des services

Les pages patient (PatientProfilePage, DoctorsListPage, etc.) appellent ces thunks sans que les endpoints aient été testés avec de vrais tokens JWT en Postman. Si la structure JSON réelle du backend diffère légèrement des types définis (patient.types.ts, doctor.types.ts), les pages seront silencieusement vides ou produiront des erreurs runtime invisibles à la compilation.

Problème 3 — Absence de reset centralisé au logout

clearPatientState est exporté et appelé dans PatientLayout lors du logout. Mais il n'existe pas de mécanisme centralisé. Quand les slices médecin, réceptionniste et admin seront ajoutés, chaque logout devra manuellement appeler clearXxxState() pour chaque slice. Risque de fuite de données entre sessions.

src/store/rootReducer.ts 🟠 Violation de méthode 
import patientReducer from '@/features/patient/patientSlice'

const rootReducer = combineReducers({
  auth: authReducer,
  ui: uiReducer,
  patient: patientReducer, // ← ajouté avant validation Phase 7
})

Le reducer patient a été ajouté dans le store alors que la Phase 7 de la méthode Ahmed Yahya (« Patient Slice — créer Redux pour médecins, profils, réservations ») n'avait pas encore été officiellement complétée et validée. Les thunks sont actifs et dispatchables depuis des composants avant que les services qu'ils appellent aient été testés avec de vrais tokens JWT.

src/store/uiSlice.ts 🟡 Anti-pattern

Problème 1 — Side effect DOM dans un reducer Redux

setLang(state, action: PayloadAction<Lang>) {
  state.lang = action.payload
  localStorage.setItem('lang', action.payload)           // ← side effect
  document.documentElement.setAttribute('lang', ...)    // ← side effect DOM
  document.documentElement.setAttribute('dir', ...)     // ← side effect DOM
}

Les reducers Redux doivent être des fonctions pures. Modifier document.documentElement et localStorage directement dans un reducer viole ce principe fondamental. Ces opérations devraient être dans un middleware ou un effet React.

Problème 2 — Guard SSR inutile

const savedLang = (typeof localStorage !== 'undefined'
  ? localStorage.getItem('lang')
  : null) as Lang | null

Ce guard typeof localStorage !== 'undefined' est un pattern de défense contre le Server-Side Rendering. CodevaSanté est une application Vite 100% client-side. localStorage est toujours disponible. Ce code est du cargo-cult sans aucune utilité dans ce contexte.

src/services/authService.ts 🟠 DANGEREUX

Problème 1 — Fonction refresh() publique crée un risque de race condition

refresh: (refreshToken: string) =>
  axios
    .post<{ access: string }>(`${BASE_URL}/auth/refresh/`, { refresh: refreshToken })
    .then((r) => r.data),
🔴 Race condition potentielle L'intercepteur dans axios.ts gère déjà le refresh automatiquement avec une queue pour les appels concurrents. Si du code appelle authService.refresh() directement en parallèle de l'intercepteur, deux appels POST /auth/refresh/ sont lancés avec le même refreshToken. Un token de rafraîchissement JWT ne peut être utilisé qu'une seule fois (rotation). Le second appel retourne 401 → logout intempestif.

Cette fonction ne devrait pas exister publiquement. La logique de refresh appartient exclusivement à axios.ts.

Problème 2 — BASE_URL dupliquée (déjà signalé en §3.1)

src/services/patientService.ts ✓ CORRECT

Fichier propre et exemplaire. Chaque méthode est une ligne, les types sont correctement importés, le commentaire sur getMyAppointments est honnête et informatif. Rien à redire sur ce fichier. C'est le standard que tous les autres services devront suivre.

src/types/models.ts 🔴 À SUPPRIMER IMMÉDIATEMENT
🔴 Types contradictoires avec les types feature-split Ce fichier définit des interfaces Doctor et Appointment qui contredisent directement doctor.types.ts et appointment.types.ts. 
// models.ts — FAUX
export interface Doctor {
  id: number
  name: string       // ← Le backend retourne full_name, pas name
  specialty: string  // ← Manquent bio, years_of_experience, available
  phone: string
  email: string
}

// doctor.types.ts — CORRECT
export interface Doctor {
  id: number
  full_name: string
  email: string
  phone: string
  specialty: string
  bio: string
  years_of_experience: number
  available: boolean
}

Si un développeur importe Doctor depuis models.ts au lieu de doctor.types.ts, TypeScript accepte le code mais les données runtime ne correspondent pas. Le bug est silencieux et difficile à diagnostiquer. Ce fichier doit être supprimé sans délai.

src/types/api.ts 🟡 Inutilisé et inexact 
export interface ApiResponse<T> {
  data: T
  message: string
  success: boolean  // ← CE CHAMP N'EXISTE PAS DANS LE BACKEND DRF
}

Le backend Django REST Framework retourne { message: string, data: T } pour les succès. Le champ success: boolean n'existe pas. Ce fichier n'est importé nulle part dans le code actuel — ce qui montre qu'il n'est pas utilisé. Il sera source de confusion si un développeur s'y réfère pour typer une réponse API.

src/routes/ProtectedRoute.tsx ✓ CORRECT

Implémentation propre et minimaliste. Deux vérifications séquentielles : isAuthenticated pour l'accès général, puis user.role pour la vérification du rôle. Les redirections vers /app/login et /app/unauthorized sont correctes. Aucun problème identifié.

src/routes/AppRoutes.tsx 🟡 À surveiller

La structure de routes imbriquées avec PatientLayout comme parent et Outlet est correcte et élégante. Cependant :

Route /patient/appointments active mais non fonctionnelle

MyAppointmentsPage est montée sur /app/patient/appointments alors que l'endpoint backend GET /api/appointments/me/ n'existe pas encore. Un patient authentifié qui navigue vers cette route verra une page "Coming Soon" sans aucun feedback clair sur la raison. La route devrait être désactivée ou rediriger vers la page d'accueil tant que l'endpoint backend n'est pas disponible.

src/routes/index.tsx 🟡 Ordre suspect 
<Route path="/*"    element={<WebsiteRoutes />} />  {/* générique en premier */}
<Route path="/app/*" element={<AppRoutes />} />      {/* spécifique en second */}

React Router v7 utilise la spécificité (non l'ordre) pour le matching. /app/login matchera correctement AppRoutes. Mais déclarer le pattern générique avant le pattern spécifique est une mauvaise pratique. Si le comportement du router change dans une version future, ce code sera silencieusement cassé. L'ordre correct : plus spécifique d'abord.

src/components/layout/PatientLayout.tsx 🔴 BLOQUANT — Build cassé
🔴 Clés de traduction inexistantes — tsc -b échoue PatientLayout référence 5 clés qui n'existent pas dans translations.ts. Avec TypeScript strict activé et TranslationKey = keyof typeof translations.fr, le build de production échoue avec des erreurs de propriété inexistante. 
const navLinks = [
  { label: t.navPatientHome,         ... },  // ← N'EXISTE PAS
  { label: t.navPatientProfile,      ... },  // ← N'EXISTE PAS
  { label: t.navPatientDoctors,      ... },  // ← N'EXISTE PAS
  { label: t.navPatientAppointments, ... },  // ← N'EXISTE PAS
]
// ...
<span>{t.navLogout}</span>                  // ← N'EXISTE PAS

Ces 5 clés doivent être ajoutées dans les deux objets (fr et ar) de translations.ts avant que le build puisse passer.

src/pages/app/patient/ (6 fichiers) 🟠 Générés hors ordre

L'ensemble des pages patient (PatientHomePage, PatientProfilePage, DoctorsListPage, DoctorDetailPage, CreateAppointmentPage, MyAppointmentsPage) a été généré et committé avant que les endpoints backend aient été validés avec de vrais tokens JWT en Postman, en violation directe de la checklist du Guide de Prompts IA Générative :

Checklist Ahmed Yahya — Points non respectés
  • ☐ Le endpoint est testé avec un vrai token JWT
  • ☐ Le payload JSON est connu
  • ☐ La réponse JSON est connue

Conséquences concrètes par page :

  • DoctorDetailPage — utilise params.id dans l'URL (/app/patient/doctors/:id) mais lit selectedDoctor depuis Redux. Le paramètre :id n'est jamais utilisé. Un accès direct par URL (lien partagé, favori) retourne une page vide. L'URL est mensongère.
  • PatientProfilePage — si GET /patients/me/ retourne une structure légèrement différente de PatientProfile, la page sera vide sans aucune erreur TypeScript.
  • CreateAppointmentPage — le champ time utilise <input type="time"> qui retourne HH:MM. L'API attend HH:MM. Compatible pour l'instant, mais aucune validation Zod n'est en place.
src/pages/website/ServicesPage.tsx + SpecialitiesPage.tsx 🟡 Fichiers morts

Les routes /services et /specialities ont été supprimées de WebsiteRoutes.tsx lors du nettoyage nav (PR #10). Ces deux fichiers existent toujours dans le repository. Ils compilent, occupent de l'espace cognitif, et peuvent être accidentellement réimportés par un développeur. À supprimer.

src/constants/translations.ts 🟠 Clés manquantes

En plus des 5 clés manquantes signalées dans PatientLayout, il convient de vérifier systématiquement toutes les clés référencées dans les pages patient nouvellement créées. Les clés candidates potentiellement absentes incluent des libellés de formulaires, états de chargement, et messages de succès/erreur spécifiques au dashboard patient.

La règle du projet est explicite : « All keys in translations.ts — no hardcoded strings ». Un passage complet via tsc -b est le seul moyen d'identifier exhaustivement les clés manquantes.

index.html 🟡 Flash RTL → LTR 
<html lang="ar" dir="rtl">

La direction RTL est codée en dur dans le HTML. uiSlice.setLang corrige document.documentElement via un side effect au démarrage dans main.tsx. Pour un utilisateur ayant sauvegardé lang=fr en localStorage, la séquence est :

  1. Page se charge en RTL (HTML statique).
  2. JavaScript s'exécute, Redux dispatch setLang('fr').
  3. DOM bascule en LTR.

Ce flash RTL→LTR est perceptible, surtout sur connexion lente. Non bloquant, mais dégradant pour l'expérience utilisateur en mode français.

5Actions immédiates obligatoires

Les actions suivantes doivent être exécutées avant de continuer tout développement. Elles sont listées dans l'ordre de priorité décroissante.

Action 1 — Ajouter la persistance auth (BLOQUANT)

Dans authSlice.ts, charger les tokens depuis localStorage à l'initialisation et les sauvegarder dans chaque reducer concerné : 

// Chargement à l'init
const storedToken   = localStorage.getItem('token')
const storedRefresh = localStorage.getItem('refreshToken')
const storedUser    = localStorage.getItem('user')

const initialState: AuthState = {
  user:            storedUser  ? JSON.parse(storedUser) : null,
  token:           storedToken  ?? null,
  refreshToken:    storedRefresh ?? null,
  isAuthenticated: !!storedToken,
}

// Dans setCredentials :
localStorage.setItem('token',        action.payload.token)
localStorage.setItem('refreshToken', action.payload.refreshToken)
localStorage.setItem('user',         JSON.stringify(action.payload.user))

// Dans setToken :
localStorage.setItem('token', action.payload)

// Dans logout :
localStorage.removeItem('token')
localStorage.removeItem('refreshToken')
localStorage.removeItem('user')

Action 3 — Supprimer src/types/models.ts (BLOQUANT)

Vérifier qu'aucune page n'importe depuis ce fichier, puis le supprimer définitivement. Les types corrects sont dans doctor.types.ts, patient.types.ts, appointment.types.ts et medical.types.ts.

Action 4 — Supprimer authService.refresh() (GRAVE)

Retirer la méthode refresh() de authService.ts. Retirer également la duplication de BASE_URL en l'exportant depuis axios.ts.

Action 5 — Supprimer les fichiers morts (MOYEN)

Supprimer ServicesPage.tsx et SpecialitiesPage.tsx du repository. Supprimer également types/api.ts tant qu'il n'est pas utilisé et que son contenu est inexact.

7Conclusion

Le projet CodevaSanté possède une architecture globalement saine : la séparation des couches (types → services → Redux → composants), l'utilisation correcte de TypeScript strict, et la logique de rafraîchissement JWT dans axios.ts démontrent une compréhension solide des bonnes pratiques.

Cependant, deux problèmes bloquants empêchent toute démonstration fonctionnelle de l'application : l'absence de persistance de l'authentification et les clés de traduction manquantes qui cassent le build TypeScript. Ces deux points doivent être traités en priorité absolue.

La principale leçon de cet audit est que la méthode Ahmed Yahya a une raison d'être : générer des pages complètes avant de valider les services et les endpoints avec de vrais tokens JWT en Postman introduit un risque de divergence silencieuse que TypeScript ne peut pas détecter. Les données JSON réelles du backend sont la seule source de vérité.

Recommandation finale Corriger les 4 points bloquants dans une PR dédiée (fix/auth-persistence-and-translations), faire passer tsc -b à zéro erreur, tester le login/logout/refresh manuellement, puis reprendre le développement selon la méthode Ahmed Yahya Phase par Phase.
